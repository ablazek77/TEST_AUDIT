## Result: FINDINGS_FOUND

Analysis of the IRRemoteFilter Arduino sketch reveals several memory safety concerns, primarily related to potential buffer overflow in the raw IR data storage path, an unchecked null pointer dereference when reading IR data, and integer type issues with timeout counters. The most significant risk is the potential for rawCodeLength to exceed the bounds of the rawCode array when storing unknown protocol data, and the use of a received IRData pointer without null checking.

## Findings

### 1. [HIGH] Potential Buffer Overflow in Raw IR Code Storage

**Description:** In storeCode(), when the protocol is UNKNOWN, rawCodeLength is set to IrReceiver.decodedIRData.rawDataPtr->rawlen - 1, and then IrReceiver.compensateAndStoreIRResultInArray(sStoredIRData.rawCode) is called. The rawCode buffer is declared as uint8_t rawCode[RAW_BUFFER_LENGTH]. If rawlen - 1 exceeds RAW_BUFFER_LENGTH, the compensateAndStoreIRResultInArray call will write beyond the bounds of rawCode, causing a heap/stack buffer overflow depending on where sStoredIRData is allocated (it is a global struct here, so BSS/data segment overflow).

### 2. [HIGH] NULL Pointer Dereference on IrReceiver.read() Return Value

**Description:** In the loop() function, IrReceiver.read() is called and its return value is passed directly to storeCode() without checking for NULL. If IrReceiver.read() returns NULL (which is possible if no data is available or an error occurs), storeCode() will dereference the null pointer at aIRReceivedData->flags, causing a null pointer dereference.

### 3. [MEDIUM] Integer Underflow in rawCodeLength Calculation

**Description:** In storeCode(), rawCodeLength is assigned the value IrReceiver.decodedIRData.rawDataPtr->rawlen - 1. The type of rawlen is typically uint16_t or similar unsigned type. If rawlen is 0 (which could occur with malformed or empty IR data), subtracting 1 from an unsigned value causes an integer underflow, resulting in a very large positive value (e.g., 65535 for uint16_t). This large value is then stored in rawCodeLength (uint8_t), causing truncation, and is also used to control the subsequent array store operation.

### 4. [MEDIUM] Signed Integer Overflow/Underflow in Timeout Counter

**Description:** The variable timeOut is declared as int (signed) and initialized to READ_TIMEOUT (30000). It is decremented by 1 each loop iteration in programState==1. The check timeOut==0 is used to detect timeout. However, if the decrement somehow skips zero (e.g., due to timing or if READ_TIMEOUT is set to a value that causes issues), timeOut will continue decrementing into negative values indefinitely, and the timeout condition will never trigger. Additionally, READ_TIMEOUT is declared as int but used as a large positive value; on platforms with 16-bit int (common on AVR), 30000 fits but is close to the signed 16-bit maximum (32767), and INIT_REPEAT_DELAY_TIME (29800) is also close to the limit.

### 5. [LOW] Unchecked Pointer Dereference of rawDataPtr

**Description:** In storeCode(), when protocol is UNKNOWN, IrReceiver.decodedIRData.rawDataPtr is dereferenced to access rawlen and then passed to compensateAndStoreIRResultInArray. There is no null check on rawDataPtr before dereferencing it.
