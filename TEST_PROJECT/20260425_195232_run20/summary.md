## Top Findings

### 1. [HIGH] Potential Out-of-Bounds Write in Raw IR Data Storage

In storeCode(), IrReceiver.compensateAndStoreIRResultInArray(sStoredIRData.rawCode) copies raw IR data into a fixed-size buffer rawCode[RAW_BUFFER_LENGTH] without first validating that rawCodeLength does not exceed RAW_BUFFER_LENGTH. A crafted IR signal with more raw entries than the buffer can hold may cause a global/stack buffer overflow, corrupting adjacent memory.

### 2. [HIGH] NULL Pointer Dereference from IrReceiver.read()

In the main loop() at programState==1, the return value of IrReceiver.read() is passed directly to storeCode() without a null check. If the library returns nullptr due to no available data or an internal error, storeCode() will dereference the null pointer when accessing aIRReceivedData->flags, causing a crash or device reset.

### 3. [MEDIUM] Integer Underflow in rawCodeLength Calculation

The expression rawlen - 1 is computed without checking that rawlen > 0. If rawlen is 0, the subtraction underflows the uint8_t assignment to 255, and the inflated length is subsequently passed to IrSender.sendRaw(), causing it to read 255 bytes from a potentially much smaller valid buffer region.

### 4. [MEDIUM] Signed Integer Underflow in Timeout Counter

The timeOut variable is a signed int decremented each loop iteration and compared with == 0. If the counter skips zero (e.g., due to multi-unit decrements or platform-specific behavior) it goes negative and the timeout never fires, potentially hanging the device in programState==1 indefinitely. On 16-bit AVR platforms, the initial value of 30000 also approaches the signed int maximum of 32767.

### 5. [LOW] Signed int Used for Non-Negative Delay and Timeout Constants

Variables such as DELAY_BETWEEN_REPEAT, NEXT_DELAY_BETWEEN_REPEAT, FIRST_DELAY_BETWEEN_REPEAT, POWER_COMMAND_DELAY, POWER_DELAY_LOCK, and READ_TIMEOUT are declared as signed int but represent non-negative durations. If any arithmetic yields a negative value, passing it to delay() (which accepts unsigned long) results in an extremely long delay, effectively hanging the device.
